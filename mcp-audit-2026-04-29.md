## P0 — do soon

### P0-1. No git history. Set up local git.
The project ships as a tarball. `.git` is absent. Loses diff-ability, rollback, and any "what changed?" instinct.
**Fix:** `cd ~/Developer/google_workspace_mcp && git init && git add . && git commit -m "Initial: v0.7.1-dev tarball"` plus a sensible `.gitignore` for `.venv/`, `token.json`, `credentials.json`, `config.json`, `logs/`, `*.bak.*`. Optional: a private remote on GitHub. **Effort: 10 min.**

### P0-2. Pick up task #235 — vendor reply parsing.
The previous holder (Finnn) explicitly handed this off as the easiest entry point. Three concrete gaps in `vendor_followups.py` + `tools/project_invoices.py::workflow_process_vendor_replies`:
1. **Multi-message thread dedup.** A vendor replies twice → MCP processes both as separate signals.
2. **Attachment extraction.** Vendor sends the missing W-9 as an attachment → MCP reads the body but doesn't pull the file into the project Drive folder.
3. **Confidence-gated promotion.** Any reply currently auto-promotes the AP row. Should be: HIGH confidence → promote, MEDIUM → leave AWAITING_INFO + flag for review.
**Effort: 1–2 days.** Tests already in `tests/test_project_invoices_tools.py`, `tests/test_vendor_replies.py` — extend with multi-msg + attachment fixtures.

### P0-3. Direct unit tests for 13 tool modules.
13 of the tool modules have **no dedicated test file**:
`gmail`, `calendar`, `drive`, `sheets`, `docs`, `tasks`, `chat`, `contacts`, `maps`, `enrichment`, `system`, `workflows`, `handoff`.
They're covered indirectly via `test_tools_registration.py`, but breakage in input validation, error paths, or schema shape would slip through. The high-line-count files (`gmail.py` 1456, `chat.py` 1765, `contacts.py` 1560, `workflows.py` 7898) are the riskiest.
**Fix:** start with input-model validation + happy-path stubs for `gmail`, `calendar`, `drive`. Pattern: mock `gservices.client(...)` and assert the input model rejects bad input + the call shape is right. **Effort: 2–3 days for the first three; rest can land incrementally.**

### P0-4. Resolve `awaiting_info.json` stale entries.
6 active entries, including 5 chat threads from the previous holder's testing. Add a `workflow_sweep_awaiting_info` that lists, ages, and bulk-clears entries older than N days. Currently nothing surfaces these except by reading the JSON.
**Effort: 2 hours.**

## P1 — this quarter

### P1-1. Split `tools/workflows.py` (7898 lines, 44 tools).
Single biggest file in the project. Maintainability cliff. Natural splits:
- `tools/workflows_gmail.py` — attachments → Drive, save-as-PDF, mail-merge, signature management
- `tools/workflows_drive.py` — bulk-share, access cleanup, folder templating
- `tools/workflows_crm.py` — contact enrichment loops, stats refresh, group ops
- `tools/workflows_calendar.py` — defrag, free/busy aggregation, recurring-event helpers
- `tools/workflows_misc.py` — what doesn't fit the above
**Effort: 1 day.** Mechanical refactor. Add a `from tools.workflows import *` shim during the transition so external imports don't break.

### P1-2. Mark network-dependent tests with `@pytest.mark.network`.
`system_doctor` reports the test suite hangs at >120s — that's because some tests hit live Google APIs. Mark them so the default suite stays fast.
**Fix:** `pytest.ini` adds `markers = network: tests requiring network`. Default `pytest` excludes `-m "not network"`. Network suite runs nightly or pre-release.
**Effort: 2 hours.**

### P1-3. Audit broad exception handlers (540 of them).
122 `except Exception:` + 418 `except Exception as e:` in production code. Many warranted (network/API calls), but some likely swallow real bugs. Sample 30 random ones; estimate the legitimate-vs-swallow ratio. If swallow-rate >10%, do a pass tightening the exception types.
**Effort: 4 hours sample + 1 day fixes if warranted.**

### P1-4. Reminder cadence + escalation (task #236).
Currently tier 1 → 2 → 3 → 4 fires on a fixed cadence. Smarter version: vendor's typical response time + day-of-week awareness + holiday calendar. If a vendor replies in <2hr 80% of the time, a 48hr nudge is too soon.
**Effort: 2–3 days.**

### P1-5. Snooze + bulk actions + visible escalation trail (task #237).
Right now AP rows have no "snooze until Mon" or "bulk-promote these 5". Trail of "we asked X on date Y, no reply, escalated Z" isn't visible to a human reviewer in Sheets — only in code logs.
**Effort: 2 days.**

### P1-6. Brand-voice corpus filter.
Today's regen pulled 702 sent emails — but the analyzer flagged "a large proportion of Google Calendar notifications and Google Drive share notifications which carry no authored text." Substantive sample was ~15–20 messages.
**Fix:** add `-from:noreply -from:calendar-notification@google.com -from:drive-shares-noreply@google.com` to the Gmail query in `refresh_brand_voice.py::_fetch_sent`. Should 5–10x the usable corpus.
**Effort: 30 min.**

### P1-7. Slack integration.
Currently `chat_*` tools cover Google Chat only. Many of the AP follow-up + executive-briefing patterns would compose with Slack identically. Worth a `tools/slack.py` parallel to `tools/chat.py`.
**Effort: 3–5 days for parity with current Chat surface.** Auth via Slack OAuth, channel + DM send/read/search.

## P2 — nice to have

### P2-1. Config schema validation.
`system_check_config` flags `_attachments_comment` as "unknown key" — but it's an inline comment, intentional. Either: (a) skip keys starting with `_`, or (b) move comments out to a sidecar `config.example.json` only. **Effort: 30 min.**

### P2-2. mypy / static type checking.
~47k lines of typed Python without static checking. Pydantic models give you input validation; mypy would catch the rest. Add to `pyproject.toml` + `make typecheck`. **Effort: 2 hours initial + cleanup of whatever it surfaces.**

### P2-3. GitHub / Linear / Asana connectors.
Engineering + project-management. Would let the AP project resolver also speak to engineering ticket systems (e.g., "this invoice is for the Linear project SF-100"). **Effort: per-integration ~3 days.**

### P2-4. Outbound webhook framework.
Currently `briefing_webhook.py` listens on `:7799` for action buttons. A general "external event → MCP action" path (e.g., GitHub webhook → file an AP entry) would unlock cross-system automation. **Effort: 2 days for a clean framework, then per-event hookups.**

### P2-5. Marketplace launch assets.
Tasks #230–#233 from the previous holder: demo GIF, ~800-word blog post, social posts, ~3-min video script. Defer unless you want this MCP to ship to the public marketplace.
